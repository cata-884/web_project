1. Introducere

1.1 Scop

Acest document reprezinta specificatia completa a cerintelor software (SRS) pentru platforma web CaT (Camping & Travel). Documentul descrie comportamentul functional si non-functional al sistemului, actorii implicati, constrangerile tehnice si scenariile de utilizare principale.

Documentul se adreseaza echipei de dezvoltare, evaluatorilor academici si oricarui stakeholder care doreste sa inteleaga scopul si limitele sistemului.

1.2 Domeniu

CaT este o aplicatie web de tip marketplace pentru descoperirea, listarea si rezervarea locatiilor de camping din Romania. Platforma conecteaza turisti cu organizatori de campinguri, ofera o harta interactiva, un sistem de recenzii cu media atasata si un panou de administrare complet.

Sistemul nu include procesarea platilor online (pretul este calculat si afisat, dar tranzactia se finalizeaza offline), si nu gestioneaza logistica transportului.

2. Descriere generala a produsului

2.1 Perspectiva produsului

CaT este o aplicatie web autonoma, accesibila dintr-un browser modern fara instalare locala. Backend-ul PHP expune un API REST consumat exclusiv de frontend-ul JavaScript SPA (Single Page Application) cu navigare intre pagini HTML statice. Datele sunt stocate intr-o baza MySQL gestionata de LAMPP.

Sistemul interactioneaza cu doua servicii externe: Google OAuth 2.0 (autentificare alternativa) si OpenStreetMap/Leaflet (afisarea hartii interactive).

2.2 Functiile principale ale produsului

  • Autentificare — inregistrare, login cu email/parola sau cont Google, gestionare profil si preferinte.
  • Catalog campinguri — listare, filtrare si cautare dupa tip, regiune, pret si rating.
  • Harta interactiva — vizualizare campinguri pe harta OSM cu filtrare si cautare locatie.
  • Rezervari — verificare disponibilitate, creare si anulare rezervari, istoric.
  • Recenzii — text + rating stele + media (imagini, audio, video), editare si stergere.
  • Colectii personale — organizare campinguri favorite in sectiuni cu nume si culori.
  • Modul organizator — aplicatie de verificare, creare si publicare campinguri.
  • Panou admin — gestionare utilizatori, aprobare campinguri, statistici, export/import date.
  • Contact — formular de mesaje intre vizitatori si echipa platformei.

2.4 Constrangeri

  • Tehnice: PHP 8.x, MySQL 8.x, Apache 2.4, JavaScript ES2020+; fara framework JS extern major (React/Vue).
  • Browser: Compatibilitate cu ultimele doua versiuni majore din Chrome, Firefox, Safari, Edge.
  • Upload media: Fisierele incarcate sunt stocate pe disc local (/uploads/); nu exista integrare CDN.
  • Plati: Sistemul nu proceseaza plati online; pretul este informativ.
  • Localizare: Interfata in limba romana (principala) cu suport i18n extensibil.
  • Reglementari: Aplicatia trebuie sa respecte GDPR pentru datele utilizatorilor din UE.

2.5 Ipoteze si dependente

  • Serverul are acces la internet pentru incarcarea Leaflet.js si Google OAuth.
  • Baza de date MySQL este accesibila pe localhost cu credentialele configurate in backend/config/app.php.
  • Directorul public/uploads/ este accesibil la scriere de catre procesul Apache.
  • Credentialele Google OAuth (client ID + secret) sunt configurate inainte de activarea autentificarii Google.

4. Cerinte non-functionale

4.1 Securitate

Autentificare si autorizare

  • NFR-SEC-01: Toate parolele sunt stocate hashuite cu bcrypt (cost ≥ 10). Niciodata in text clar.
  • NFR-SEC-02: Sesiunile sunt gestionate prin Bearer Tokens cu expirare configurabila (implicit 30 de zile). Token-ul este invalidat la logout pe server.
  • NFR-SEC-03: Toate endpoint-urile care modifica date verifica rolul utilizatorului autentificat. Accesul neautorizat returneaza HTTP 401/403.
  • NFR-SEC-04: Operatiunile pe resurse proprii (editare camping, stergere recenzie) verifica ownership-ul in model, nu doar in controller.

Validarea datelor si protectie impotriva atacurilor

  • NFR-SEC-05: Toate interogarile SQL folosesc prepared statements PDO pentru prevenirea SQL Injection (OWASP A03).
  • NFR-SEC-06: Datele afisate in HTML sunt escapate pentru prevenirea XSS (OWASP A03). Template strings JS folosesc textContent unde este posibil.
  • NFR-SEC-07: Fisierele incarcate sunt validate dupa tip MIME si extensie. Nu se executa fisiere incarcate de utilizatori.
  • NFR-SEC-08: Header-ele HTTP de securitate sunt configurate pe server: X-Content-Type-Options: nosniff, X-Frame-Options: DENY.
  • NFR-SEC-09: Utilizatorii cu ban activ primesc HTTP 403 la orice request autentificat.

GDPR si date personale

  • NFR-SEC-10: Utilizatorii pot solicita stergerea contului. Stergerea cascadeaza rezervarile si recenziile asociate.
  • NFR-SEC-11: Datele de sesiune expirate sunt curatate periodic prin job de cleanup.

4.2 Performanta

  • NFR-PERF-01: Pagina de catalog (/api/campings) trebuie sa raspunda in mai putin de 500 ms pentru maximum 100 de rezultate pe pagina.
  • NFR-PERF-02: Endpoint-ul de harta (/api/campings/map) returneaza cel mult campurile necesare marcatorilor (id, lat, lng, name, type, rating) pentru a minimiza payload-ul.
  • NFR-PERF-03: Imaginile incarcate de utilizatori nu sunt redimensionate automat in faza actuala; se recomanda limitarea la 10 MB/fisier la nivel de server.
  • NFR-PERF-04: Paginile HTML sunt servite de Apache cu compresie gzip activata.
  • NFR-PERF-05: Interogarile de cautare folosesc indecsi pe coloanele approval_status, type, region si price_per_night.

4.3 Scalabilitate

  • NFR-SCAL-01: Arhitectura stateless (token-based) permite adaugarea de instante Apache fara sincronizare de sesiune.
  • NFR-SCAL-02: Schema bazei de date poate suporta pana la 100.000 de campinguri si 1.000.000 de rezervari fara modificari structurale, cu conditia indexarii corecte.
  • NFR-SCAL-03: Upload-ul de media este izolat intr-un modul MediaController, permitand migrarea viitoare catre un serviciu de stocare extern (S3-compatibil) fara refactoring major.

4.4 Disponibilitate si fiabilitate

  • NFR-AVAIL-01: Sistemul trebuie sa fie disponibil 99% din timp in intervalul de lucru (08:00–24:00).
  • NFR-AVAIL-02: Erorile API (5xx) trebuie sa returneze un JSON cu campul error descriptiv. Frontend-ul afiseaza un toast de eroare, nu o pagina alba.
  • NFR-AVAIL-03: Baza de date beneficiaza de backup zilnic al dump-ului MySQL.

4.5 Utilizabilitate (Usability)

  • NFR-USE-01: Interfata este responsive si functionala pe dispozitive mobile (≥ 360 px latime) si desktop.
  • NFR-USE-02: Notificarile catre utilizator (succes, eroare, avertisment) sunt afisate prin componenta toast non-blocanta, nu prin alert() nativ.
  • NFR-USE-03: Actiunile distructive (stergere recenzie, anulare rezervare) necesita confirmare explicita prin dialog custom.
  • NFR-USE-04: Interfata suporta doua teme (light / dark) commutabile din panoul de cont.
  • NFR-USE-05: Structura de titluri HTML respecta ierarhia h1 → h2 → h3 pentru accesibilitate.
  • NFR-USE-06: Textele sunt internalizate prin sistemul i18n cu fisiere de traducere JSON, permitand adaugarea de noi limbi fara modificari de template.

4.6 Portabilitate si mentenabilitate

  • NFR-PORT-01: Backend-ul ruleaza pe orice server cu PHP 8+ si MySQL 8+ (XAMPP, LAMPP, hosting standard).
  • NFR-PORT-02: Frontend-ul este compus din fisiere HTML, CSS si JS statice fara build step, deployabile direct pe orice server web.
  • NFR-MAINT-01: Migrarile de schema SQL sunt versionizate in backend/db/migrations/ si rulate in ordine numerica.
  • NFR-MAINT-02: Codul backend respecta separarea Controller–Model–Router; logica de business nu exista in router sau fisiere de configurare.

8. Use Cases principale

UC-01 — Inregistrare utilizator nou
Actor principal Vizitator
Preconditii Utilizatorul nu are cont existent cu acelasi email sau username.
Flux principal
  1. Vizitatorul acceseaza /pages/register.html.
  2. Completeaza formularul: username, email, parola, confirmare parola.
  3. Trimite formularul (POST /api/auth/register).
  4. Sistemul valideaza campurile si unicitatea.
  5. Sistemul creeaza contul, genereaza un Bearer Token si il returneaza.
  6. Frontend-ul salveaza token-ul si redirecteaza la /pages/account/account.html.
Flux alternativ Daca email-ul sau username-ul exista deja, se afiseaza toast de eroare si formularul ramane deschis.
Postconditii Utilizatorul are cont activ si este autentificat.
UC-02 — Rezervare camping
Actor principal Utilizator autentificat
Preconditii Utilizatorul este autentificat. Camping-ul este aprobat si publicat.
Flux principal
  1. Utilizatorul acceseaza pagina de detaliu a camping-ului.
  2. Selecteaza data check-in, check-out si numarul de oaspeti.
  3. Sistemul calculeaza pretul total si il afiseaza in timp real.
  4. Utilizatorul apasa "Rezerva acum" (POST /api/bookings).
  5. Sistemul verifica disponibilitatea intervalului ales.
  6. Sistemul creeaza rezervarea cu status pending.
  7. Se afiseaza toast de succes si utilizatorul este redirectat la contul sau.
Flux alternativ A Daca datele nu sunt selectate: toast de avertisment, rezervarea nu este trimisa.
Flux alternativ B Daca camping-ul nu este disponibil in intervalul ales: API returneaza eroare, se afiseaza toast de eroare.
Postconditii Rezervarea apare in istoricul utilizatorului cu status pending.
UC-03 — Adaugare recenzie cu media
Actor principal Utilizator autentificat
Preconditii Utilizatorul este autentificat si nu a lasat deja o recenzie la acest camping.
Flux principal
  1. Utilizatorul naviga la pagina de detaliu camping.
  2. Sectiunea "Scrie o recenzie" este vizibila (afisata de JS dupa verificarea eligibilitatii).
  3. Selecteaza ratingul (1–5 stele) prin click pe stele.
  4. Scrie textul recenziei in textarea.
  5. Optionalni: selecteaza fisiere media (imagini, audio, video).
  6. Apasa "Trimite Recenzia" (POST /api/campings/{id}/reviews).
  7. Daca exista fisiere: upload multipart pentru fiecare (POST /api/reviews/{id}/media).
  8. Pagina se reincarca automat; recenzia apare in lista.
  9. Ratingul mediu al camping-ului este actualizat.
Flux alternativ Daca niciun rating nu este selectat: toast de avertisment "Selecteaza o nota!"
Postconditii Recenzia este publicata. Ratingul mediu al camping-ului este recalculat.
UC-04 — Publicare camping de catre organizator
Actor principal Organizator
Preconditii Utilizatorul are rolul organizer (aplicatie aprobata de admin).
Flux principal
  1. Organizatorul acceseaza tab-ul "Organizator CaT" din contul sau.
  2. Completeaza formularul de creare camping: nume, descriere, tip, adresa, coordonate, pret, capacitate, facilitati, mediu.
  3. Incarca imagini pentru galerie.
  4. Trimite camping-ul (POST /api/campings).
  5. Camping-ul primeste statusul pending.
  6. Adminul aproba camping-ul (PATCH /api/admin/campings/{id}/approve).
  7. Camping-ul devine vizibil in catalog si pe harta.
Flux alternativ Admin respinge cu feedback → organizatorul corecteaza → retrimite (POST /api/campings/{id}/resubmit).
Postconditii Camping-ul aprobat este vizibil publicului.
UC-05 — Banare utilizator de catre admin
Actor principal Administrator
Preconditii Adminul este autentificat. Utilizatorul tinta exista si nu este deja banat.
Flux principal
  1. Adminul acceseaza tab-ul "Gestionare utilizatori" din panoul admin.
  2. Cauta si selecteaza utilizatorul tinta.
  3. Apasa "Baneaza" si completeaza motivul si durata (sau permanent).
  4. Trimite cererea (POST /api/admin/users/{id}/ban).
  5. Sistemul creeaza inregistrarea ban si invalideaza toate sesiunile active ale utilizatorului.
  6. La urmatorul request al utilizatorului banat, API returneaza HTTP 403.
Flux alternativ Adminul poate ridica interdictia (POST /api/admin/users/{id}/unban).
Postconditii Utilizatorul nu mai poate efectua actiuni autentificate pana la expirarea ban-ului.
UC-06 — Export date de catre admin
Actor principal Administrator
Preconditii Adminul este autentificat cu rol admin.
Flux principal
  1. Adminul acceseaza sectiunea "Export / Import" din panoul admin.
  2. Selecteaza tipul de date (campinguri, rezervari, recenzii, utilizatori).
  3. Alege formatul dorit (CSV sau JSON).
  4. Apasa butonul de export.
  5. Browser-ul descarca fisierul generat de server (/api/admin/export/{type}.{format}).
Flux alternativ —
Postconditii Fisierul cu datele solicitate este salvat pe dispozitivul adminului.
